Collision / Mouse Follow

Exercise #1

Challenge: Max Acceleration and Sprite Rotation

The guidelines for this challenge were to give the image a maximum acceleration as well as drawing a sprite and giving it an orientation.
I implemented the former using the interpolate function given to us and a variable, accel, initially set to 0.03. I'm not sure if this was the intended way but it makes it look good, so I kept it.
I implemented the latter first by loading an image of my friends cat. I then gave it a base orientation depending on the direction it normally faced.
At each time step, I calculated the angle between the image direction and the current velocity and rotated the image that much.
It leads to a fairly smooth turn as long as you don't move the mouse too much at once. This concern, however, is addressed in a later part of this homework.

Particle Systems

Exercise #2

Challenge: Delete Old Particles and Change Color

The guidelines for this challenge were to delete particles that have been around too long, change the color of the particles over time, and change the color as a function of the bounce.
Deleting particles was straight forward: turn the position, velocity, life, and color arrays into ArrayLists and append / remove when needed.
The same effect could have been achieved using a circular queue structure, and would likely be much faster than ArrayLists, however, I did not have the time to do that. It might be something I revisit in the future.
Changing the color over time and as a function of the bounce was also straight forward. The latter is demonstrated in the above video.
The particles change color based on the change in velocity between time steps. If the ball hits them hard, they change faster than just bouncing off the ground or wall.
I also implemented the color change over time, but it is currently commented out because I thought this result looked really pretty!

Boids / TTC

Exercise #3

Challenge: Max Force / Velocity, Initial-Collision Handling, and Static Obstacles

The guidelines for this challenge were to cap the agents' force and velocity, handle starting position collisions, and add static obstacles for the agents to avoid.
I achieved the first by creating maxVel and maxAcc variables and if the length of acceleration or velocity vectors violated those numbers, normalize the vector and multiply it by the corresponding max.
The second was achieved purely by making them be affected by a separation force.
The third was achieved by having an array of obstacles with random positions and performing TTC on them for all the agents.
I also achieved this challenge by having the agents not cross the wall boundaries. My initial thought on how to do this was to cast rays in an arc and have a force applied to the agent based on the TTC for each of those rays.
This turned out to not be a great solution, however, I, akin to the first exercise, found a better way to do it in a later exercise.
This worked very similarily to the agents avoiding the big agents once they reached their goal.
In addition to those, I added attraction and alignment forces to the agents as well as attracting the agents to the user's mouse position when they left click, and repelling them from the position when they right click.

Graphs:

Challenge: Cycle Graph, DLS, and IDDFS

The guidelines for this challenge were to create a graph with cycles and see how DFS handles it, make a depth limited search algorithm, and make an iterative deepening depth first search algorithm.
The first was done by creating graph 3. Depth first search handled this graph because of the visited list, so I created another DFS without the visited list and it predictably failed.
A thing to note about this is that DFS would not fail if used on graph 4 which is functionally similar to graph 3, however, the nodes are added in a different order.
Depth Limited Search was done recursively because it was the way I knew how to do it.
Iterative Deepening Depth First Search was implemented on top of the DLS code already written and, predictably, does much better than DLS and DFS.

Planning

Exercise #5

Challenge: Rectangle List and Dragging Obstacles

The guidelines for this challenge were to add support for a list of rectangles and let the user click and drag them.
The first was achieved easily using an ArrayList and replacing most of the instances of passing a single recangle object through with that list.
The second was achieved by using the mouseDragged() function. It checks that the user clicks inside a box and saves the current mouse position if they had.
Then, when the draw function is called, it checks to see if the mouse is being dragged. If it is, it reads the current mouse position, gets the vector from last to current, and moves the box's top left by that much.
The current implementation only reconnects neighbors once the box is moved, however, I was not sure if it should redo the entire PRM algorithm, so I put that in as well, just commented out.

Kiwi Prison Experiment

Bonus

Challenge: The Entire Thing

This little extra project took aspects from the first three exercises and combined them into like a weeks worth of work.
I definitely spent way too much time on this and its not even as good as I want it to be, but I had to stop at some point.
This is my first foray into creating 3D animations so it was really fun to learn my way through the processing documentation!
At the fundamental level, it is very similar to the TTC exercise. Each kiwi is it's own agent with velocity, acceleration, and position.
The forces being applied on the kiwis are attraction to other kiwis, separation from their friends, separation from the moais in the corners (non-infected),
repulsion from the fences, alignment, and attraction to the moais (infected).
Additionally, non-infected kiwis are much more repulsed by infected kiwis and clicking the mouse either attracts (left) or repels (right) the kiwis from current camera position.
Each timestep, there is a 0.05% chance for a kiwi to get infected. Once they do, they run to the closest moai and get cleansed.
I think this leads to a cool little simulation of kiwis running around. The wall forces improve on the ones present in the TTC simulation by adding a strong separation force from the walls.
In addition to the forces, this applies concepts from the particle systems exercise, as well as the mouse follow exercise.
Representing the particle systems, each moai has a cone of white particles coming out of it that disappear over the roughly second that they are around, giving the affect of mist. (kind of)
Additionally, each kiwi rotates towards its velocity as it is walking. I improved on the previous rotation by applying a technique that Professor Guy told me about in which the angle is changed with this equation:
agentDir[id] = agentDir[id]*t+(1-t)*atan2(agentVel[id].x, agentVel[id].z);
The equation takes the previous direction and changes it only slightly based on the t value that makes it look okay. The t value is also somewhat based on the current velocity of the kiwi, however, I did not perfect this.
Despite this, the kiwis are still quite prone to spinning, but, honestly, I think its kinda cute so I'm leaving it.
Last two things about this one is that is has a camera that can snap onto kiwis locations and follow them (c) and also cancel that (shift+c)
and I remade the Vec2 library into a Vec3 library with a couple more functions :)
It was a fun project and I can't wait for the next ones!
